OxCaml Embeds

Embed runnable, checkable, and toplevel OxCaml examples in ordinary HTML pages. Start with the basic <oxcaml> tag, then choose an execution mode, optional JavaScript mounting, output behavior, color scheme, and styling hooks.

Quick Start

Add oxcaml-embed.js to the page and write examples as <oxcaml> tags. With no mode attribute, the snippet runs as a small program.

Run a Program

<oxcaml>
let greeting name =
  "Hello, " ^ name

let () =
  print_endline (greeting "OxCaml")
</oxcaml>
let greeting name = "Hello, " ^ name let () = print_endline (greeting "OxCaml")

Use Toplevel Phrases

Add utop for examples that should behave like toplevel phrases. End each phrase with ;;. 

<oxcaml utop>
let language = "OxCaml";;
let answer = 40 + 2;;
Printf.sprintf "%s: %d" language answer;;
</oxcaml>
let language = "OxCaml";; let answer = 40 + 2;; Printf.sprintf "%s: %d" language answer;;

Load the embed script once on the page, usually just before </body>. 

<script src="./oxcaml-embed.js"></script>

After the script loads, each tag becomes an editor with a status and output panel. User edits are saved in localStorage and the reset button restores the original source.

Execution Modes

A widget normally runs the source as a program. Add check when an example should only type-check, or utop when the source should behave like toplevel phrases. In utop mode, terminate phrases with ;; and the transcript shows evaluated values and inferred types for each phrase.

ModeMarkupUse it for
Run<oxcaml>Small programs that print output or emit rendered HTML.
Check<oxcaml check>Type-checking examples, expected errors, and snippets without side effects.
Utop<oxcaml utop>Interactive lesson examples where each phrase should show its value and type.
 
<oxcaml utop data-filename="lesson_values.ml">
let language = "OxCaml";;
let answer = 40 + 2;;
Printf.sprintf "%s: %d" language answer;;
</oxcaml>
let language = "OxCaml";; let answer = 40 + 2;; Printf.sprintf "%s: %d" language answer;;

Programmatic mounts use the same mode names: 

window.OxCamlPlayground.mount(element, {
  mode: "utop",
  source: "let x = 1 + 2;;",
});

Programmatic Mounting

Use markup when the source belongs in the HTML document. Use mount when the source comes from JavaScript, an exercise database, or another dynamic source. If the target is an ordinary DOM element, the target stays in the document and the generated widget replaces its children. 

<div id="programmatic-example" class="compact-example blue-example"></div>

<script>
  window.OxCamlPlayground.load.then((api) => {
    api.mount(document.querySelector("#programmatic-example"), {
      mode: "run",
      filename: "programmatic_example.ml",
      source: `let message = "mounted from JavaScript"

let () =
  print_endline message`,
    });
  });
</script>

When the target is not an <oxcaml> tag, mount requires source. It accepts the same option names as the markup API: mode, filename, emptyOutput, runTrigger, and storageKey. CSS custom properties on the target element inherit into the generated widget. 

const editor = api.mount(target, {
  mode: "check",
  filename: "exercise.ml",
  emptyOutput: "hide",
  runTrigger: "manual-after-initial",
  storageKey: "chapter-2/exercise-4",
  source: "let add_tax cents = cents + (cents / 10)",
});

The same mount function can also replace an existing <oxcaml> tag: 

const editor = api.mount(document.querySelector("oxcaml[data-example='intro']"), {
  mode: "utop",
});

Output Panel

By default, completed widgets keep the output panel visible and show (no output) when the program or check produced no transcript. Add data-oxcaml-empty-output="hide" to remove the output panel only when there is no transcript and no inferred-type panel.

Diagnostics, warnings, exceptions, stdout, rendered HTML output, and inferred types still show the output panel. The panel also remains visible while the widget is loading or running. 

<oxcaml check data-oxcaml-empty-output="hide">
let add_tax cents =
  cents + (cents / 10)
</oxcaml>

Programmatic mounts use the same setting name: 

window.OxCamlPlayground.mount(element, {
  mode: "check",
  emptyOutput: "hide",
  source: "let add_tax cents = cents + (cents / 10)",
});

Run Trigger

Widgets run or check as you type by default. Add data-oxcaml-run-trigger="manual" when an example should wait for the reader. Manual widgets show a mode-specific button with the keyboard shortcut in an immediate tooltip. Use manual-after-initial to run or check once on initial load, then require the button after edits. 

<oxcaml check data-oxcaml-run-trigger="manual">
let add_tax cents =
  cents + (cents / 10)
</oxcaml>
let add_tax cents = cents + (cents / 10) 
<oxcaml data-oxcaml-run-trigger="manual">
let message = "Run me with the button"
let () = print_endline message
</oxcaml>
let message = "Run me with the button" let () = print_endline message 
<oxcaml check data-oxcaml-run-trigger="manual-after-initial">
let discounted cents =
  cents - (cents / 5)
</oxcaml>
let discounted cents = cents - (cents / 5)

Programmatic mounts use runTrigger: "manual" or runTrigger: "manual-after-initial". The shortcut is ⌘↵ on macOS and Ctrl+Enter elsewhere when the editor has focus.

Color Scheme

Every widget reads --oxcaml-color-scheme from the cascade. Valid values are auto, light, and dark. Auto follows prefers-color-scheme.

Markup

Set --oxcaml-color-scheme on the original <oxcaml> tag, a class, or any ancestor. The generated widget root preserves inline style and class attributes, so the custom property keeps cascading after mounting. 

<oxcaml style="--oxcaml-color-scheme: dark">...</oxcaml>
let mode = "forced light" let () = print_endline mode let mode = "forced dark" let () = print_endline mode

JavaScript

The wrapper exposes window.OxCamlPlayground.load, a promise for the module, and then installs the public API on window.OxCamlPlayground. 

await window.OxCamlPlayground.load;

// Update every mounted widget and make future widgets inherit the same scheme.
document.documentElement.style.setProperty("--oxcaml-color-scheme", "dark");
document.documentElement.style.setProperty("--oxcaml-color-scheme", "light");
document.documentElement.style.setProperty("--oxcaml-color-scheme", "auto");

Set the same custom property on a generated .oxcaml-embed root or container element to update a smaller scope. 

const editor = window.OxCamlPlayground.mount(element, {
  source: "print_endline \"hello\"",
});

editor.root.style.setProperty("--oxcaml-color-scheme", "light");
document
  .querySelector("#chapter-2")
  .style.setProperty("--oxcaml-color-scheme", "dark");

The buttons above set --oxcaml-color-scheme on this page, so they are also a working example of global CSS control.

Preserved Attributes

The generated widget root keeps useful presentation attributes from the original <oxcaml> tag:

  • id for anchors and direct selection.
  • class for reusable styling variants.
  • style for per-widget custom properties.
  • title, aria-*, and supported data-* attributes.
<oxcaml
  id="intro-example"
  class="compact-example flat-example"
  data-topic="functions"
  aria-label="Runnable square function"
  style="--oxcaml-editor-min-height: 5.5rem">
let square x = x * x
</oxcaml>
let square x = x * x let () = Printf.printf "%d\n" (square 9)

CSS Custom Properties

Set variables on an ancestor to style a whole article, on a class to create a named variant, or inline on one <oxcaml> tag. Variables inherit into widgets, and widget-local variables override page-level values. 

.tutorial-page {
  --oxcaml-accent: #0f766e;
  --oxcaml-editor-min-height: 8rem;
}

.tutorial-page .warning-example {
  --oxcaml-accent: #b45309;
  --syntax-string: #166534;
}
